软件工程定律笔记

2026-06-29 1 分钟阅读

Source: https://lawsofsoftwareengineering.com/

软件系统的架构往往反映了该组织的结构或团队架构。

过早优化是万恶之源

只要有足够数量的 API 用户，你系统的所有可观察行为都会被某些人所关注/利用。(各种边界都会被庞大的用户群体触达)

把代码弄得比你接手时更好

YAGNI (You Aren’t Gonna Need It)，必要时再去添加功能

在软件项目已经延迟的情况下再增加人手，只会让项目更加延期。（新来的人需要理解，沟通引入额外成本）

能够良好运行的复杂系统，往往源于能够良好运行的简单系统（先做出MVP，再以此为基础增加功能)

所有非平凡抽象概念，在某种程度上都存在泄漏的问题。 （这导致有时开发者不得不在边缘情况下考虑被抽象隐藏的内部实现细节）

每个应用程序都包含一定的不可简化复杂性，这种复杂性是无法被消除的，只能进行转移。（好的设计是内部隐藏而非消除这些复杂性，从而使用户的操作变得简单）

分布式系统只能同时保证两个特性：一致性、可用性和分区容错性。【一致性（所有节点都能看到相同的数据）、可用性（每个请求都能得到响应）以及分区容错性（系统在网络故障的情况下仍能正常运行）。】

那些规模较小但成功的系统，往往会被那些设计过度、功能臃肿的替代系统所取代。（扩展时要保持谨慎，即便有之前成功的版本，新功能的复杂性也可能会是巨大的）

分布式计算中一些错误的假设，（要考虑实际场景的各种偶然情况）
该网络非常可靠；延迟为零；带宽是无限的； 该网络是安全的；
拓扑结构不会发生变化；有一位管理员；运输成本为零；该网络是同质化的；

每当你修改一个复杂的系统时，难免会遇到意外的情况。

每个程序都试图不断扩展自己的能力，直到能够读取邮件为止。（APP人均可以网贷/短视频）

一个人能够维持的稳定关系数量大约有限，约为 150 个左右。(5 代表亲密关系，~15 代表值得信赖的合作伙伴，50 代表紧密的工作关系，而150 则代表稳定的社会联系。)

随着群体规模的增加，个体的生产力会下降。(更多沟通成本，觉得别人会做从而自己不做)
